**DNS** (Domain Name System) — это распределённая иерархическая система, обеспечивающая преобразование (разрешение) человекочитаемых доменных имён (например, `example.com`) в машинные IP-адреса (например, `93.184.216.34`). Являясь одной из ключевых инфраструктур Интернета, DNS выступает в роли «телефонной книги», позволяя пользователям обращаться к ресурсам по запоминающимся именам, а не по числовым адресам. Протокол определён в серии RFC (1034, 1035 и последующие).

---

### 1. Назначение и фундаментальные принципы

Основная задача DNS — **разрешение имён**, то есть ответ на вопрос: «Какой IP-адрес соответствует данному доменному имени?» Однако DNS также используется для других целей: определения почтовых серверов для домена (MX-записи), псевдонимов (CNAME), проверки подлинности электронной почты (TXT-записи) и многого другого.

**Ключевые характеристики:**

- **Распределённая иерархия:** База данных DNS не хранится в одном месте, а распределена между миллионами серверов по всему миру. Каждый сервер отвечает только за свою зону ответственности. Это обеспечивает отказоустойчивость и масштабируемость.
- **Децентрализация:** Ни одна организация не управляет всей системой целиком. Ответственность делегируется от корня к верхним уровням, а затем к владельцам конкретных доменов.
- **Кэширование:** Для снижения нагрузки на корневые серверы и ускорения работы, результаты DNS-запросов кэшируются на всех уровнях — от операционной системы клиента до DNS-резолверов провайдеров. Это приводит к тому, что изменения в DNS распространяются не мгновенно, а в течение времени жизни записи (TTL, Time To Live).
- **Работа поверх UDP и TCP:** Для обычных запросов используется **UDP** (порт 53) — это быстро и не требует установки соединения. Однако для больших ответов (например, содержащих множество записей), зонных передач (трансфер всего домена) и запросов с расширениями (EDNS) используется **TCP** (порт 53) из-за его надёжности и способности передавать большие объёмы данных.
- **Иерархическое пространство имён:** DNS представляет собой дерево, корнем которого является точка (`.`). Это дерево делится на зоны, иерархически делегируемые сверху вниз.

### 2. Структура и иерархия DNS

Пространство имён DNS строится по принципу иерархического дерева. Доменное имя читается справа налево, от самого высокого уровня к самому низкому.

**Иерархия (сверху вниз):**

```text
. (Корневой уровень) — обозначается точкой, но обычно не отображается в адресной строке.
 │
 ├── .com (Зона верхнего уровня — TLD, Top-Level Domain)
 │    └── example.com (Вторичный домен второго уровня, SLD)
 │         └── www.example.com (Поддомен третьего уровня — запись A или CNAME)
 │         └── mail.example.com (Поддомен для почты)
 ├── .org
 ├── .ru (Национальный TLD)
 │    └── yandex.ru
 └── ... (более 1500 TLD)
```

**Уровни иерархии:**

1.  **Корневой уровень (Root Level):** Обозначается пустой строкой или точкой `.`. Корневые серверы (13 логических групп, обозначаемых буквами A–M) являются отправной точкой для любого DNS-запроса. Они не хранят информацию о доменах, но знают, где находятся серверы для каждой зоны верхнего уровня (TLD).
2.  **Зоны верхнего уровня (Top-Level Domains, TLD):** Это следующий уровень иерархии. Делятся на:
    - **Общие (gTLD):** `.com`, `.org`, `.net`, `.edu`, `.gov`.
    - **Национальные (ccTLD):** `.ru`, `.us`, `.uk`, `.de`.
    - **Новые (new gTLD):** `.tech`, `.blog`, `.online`.
3.  **Домены второго уровня (Second-Level Domains, SLD):** Регистрируются конечными пользователями в рамках TLD (например, `google.com`, `yandex.ru`).
4.  **Поддомены (Subdomains):** Создаются владельцем домена для организации структуры своих сервисов (например, `mail.google.com`, `api.example.com`).

### 3. Типы DNS-записей (Resource Records, RR)

DNS-записи — это базовые элементы базы данных DNS, каждый из которых содержит определённый тип информации о домене.

| Тип записи | Назначение | Пример использования | Пояснение |
| :--- | :--- | :--- | :--- |
| **A** | IPv4-адрес узла | `example.com. → 93.184.216.34` | Самая распространённая запись, связывает домен с 32-битным IPv4-адресом. |
| **AAAA** | IPv6-адрес узла | `example.com. → 2606:2800:220:1:248:1893:25c8:1946` | Аналог A-записи для 128-битных IPv6-адресов. |
| **CNAME** | Каноническое имя (псевдоним) | `www.example.com. → example.com.` | Создаёт псевдоним для другого доменного имени. Позволяет указать несколько имён (например, `www`, `ftp`) на один основной домен. **Важно:** CNAME-запись не может существовать вместе с другими записями (кроме DNSSEC) для одного и того же имени. |
| **MX** | Почтовый обменник (Mail Exchange) | `example.com. → mail.example.com. (приоритет 10)` | Указывает сервер, который принимает электронную почту для домена. Приоритет определяет порядок использования: чем меньше число, тем выше приоритет. |
| **NS** | Авторитарный сервер имён (Name Server) | `example.com. → ns1.example.com.` | Делегирует зону конкретному DNS-серверу, указывая, кто является авторитарным источником информации для этого домена. |
| **TXT** | Произвольный текстовый блок | `example.com. → "v=spf1 include:_spf.google.com ~all"` | Используется для самых разных целей, в основном для подтверждения владения доменом и настройки политик безопасности почты (SPF, DKIM, DMARC). |
| **PTR** | Указатель (обратная запись) | `34.216.184.93.in-addr.arpa. → example.com.` | Используется для обратного преобразования IP-адреса в доменное имя (обратный DNS). Этот процесс важен для борьбы со спамом. |
| **SOA** | Начало авторитарной зоны | Содержит параметры всей зоны: серийный номер (обновляется при изменениях), время обновления и повторных попыток для вторичных серверов, а также TTL по умолчанию. | Обязательная запись для любой зоны, описывает её базовые свойства. |

### 4. Процесс разрешения доменного имени (Рекурсивный запрос)

Процесс преобразования домена в IP-адрес проходит несколько этапов. Рассмотрим классический сценарий, когда клиент (например, браузер на ПК) запрашивает `www.example.com`.

1.  **Браузер (Клиент):** Проверяет локальный кэш DNS. Если записи нет, отправляет запрос на локальный DNS-резолвер (обычно это сервер провайдера или публичный сервер, например, `8.8.8.8`).
2.  **Локальный DNS-резолвер:** Проверяет свой кэш. Если записи нет, он начинает **рекурсивный** запрос от имени клиента:
    - **Запрос к корневому серверу (Root):** Резолвер спрашивает у одного из корневых серверов: «Где найти TLD-сервер для `.com`?». Корневой сервер отвечает и предоставляет список NS-записей для зоны `.com`.
    - **Запрос к TLD-серверу (`.com`):** Резолвер обращается к TLD-серверу `.com` с вопросом: «Где найти авторитарные серверы для `example.com`?». TLD-сервер предоставляет список NS-записей для `example.com`.
    - **Запрос к авторитарному серверу (`example.com`):** Резолвер обращается к авторитарному серверу `example.com` и запрашивает A-запись для имени `www`.
3.  **Ответ:** Авторитарный сервер возвращает IP-адрес (A-запись). Локальный резолвер кэширует ответ и отправляет его клиенту.
4.  **Клиент:** Получает IP-адрес (например, `93.184.216.34`) и устанавливает соединение с веб-сервером.

**ASCII-схема процесса:**

```text
Клиент (ПК)                Локальный DNS-резолвер          Корневой сервер        TLD-сервер (.com)       Авторитарный сервер
    │                              │                             │                    │                   (example.com)
    │  1. Запрос: www.example.com? │                             │                    │                   │
    │─────────────────────────────>│                             │                    │                   │
    │                              │  2. Где .com?               │                    │                   │
    │                              │────────────────────────────>│                    │                   │
    │                              │  3. Серверы для .com        │                    │                   │
    │                              │<────────────────────────────│                    │                   │
    │                              │  4. Где example.com?        │                    │                   │
    │                              │────────────────────────────>│                    │                   │
    │                              │  5. Серверы для example.com │                    │                   │
    │                              │<────────────────────────────│                    │                   │
    │                              │  6. A-запись для www?       │                    │                   │
    │                              │─────────────────────────────────────────────────────────────────>│
    │                              │  7. Ответ: 93.184.216.34    │                    │                   │
    │                              │<─────────────────────────────────────────────────────────────────│
    │  8. Ответ: 93.184.216.34     │                             │                    │                   │
    │<─────────────────────────────│                             │                    │                   │
```

### 5. Авторитарные и кэширующие DNS-серверы

В экосистеме DNS существует чёткое разделение ролей для серверов:

- **Авторитарный DNS-сервер (Authoritative Name Server):** Это «источник истины» для конкретной DNS-зоны. Он хранит официальные записи для домена (A, MX, NS и т.д.) и отвечает на запросы о них. Владелец домена управляет своим авторитарным сервером (обычно через панель управления хостингом). Именно на него ссылаются NS-записи домена.
- **Кэширующий DNS-сервер (Recursive Resolver):** Это сервер, который выполняет всю «грязную» работу по поиску ответа от авторитарных серверов от имени клиента. Он не является источником данных, а только рекурсивно обходит иерархию, кэшируя полученные ответы для ускорения будущих запросов. К этому типу относятся публичные серверы (Google Public DNS `8.8.8.8`, Cloudflare `1.1.1.1`) и серверы интернет-провайдеров.

### 6. Утилита `nslookup` (Name Server Lookup)

`nslookup` — это классическое консольное приложение для диагностики и исследования DNS-инфраструктуры. Она позволяет отправлять запросы к DNS-серверам и получать информацию о записях.

#### 6.1. Синтаксис и базовое использование

```bash
# Базовый запрос (использует DNS-сервер по умолчанию, установленный в системе)
nslookup example.com

# Запрос с указанием конкретного DNS-сервера (например, для обхода кэша локального резолвера)
nslookup example.com 8.8.8.8

# Просмотр конкретного типа записи
nslookup -type=A example.com
nslookup -type=MX example.com
nslookup -type=NS example.com
```

#### 6.2. Интерактивный режим

`nslookup` можно запустить в интерактивном режиме для выполнения нескольких последовательных запросов.

```bash
nslookup
> server 8.8.8.8          # Установить сервер для запросов
> set type=MX             # Установить тип запроса (MX, A, NS, SOA, CNAME, etc.)
> example.com             # Выполнить запрос
> exit                    # Выйти из программы
```

#### 6.3. Интерпретация результатов

Пример ответа `nslookup -type=A example.com`:

```text
Server:         8.8.8.8                # DNS-сервер, которому был отправлен запрос
Address:        8.8.8.8#53             # Адрес и порт этого сервера

Non-authoritative answer:              # Ответ получен не от авторитарного сервера (из кэша резолвера)
Name:    example.com                   # Запрошенное доменное имя
Address: 93.184.216.34                 # Найденный IPv4-адрес
```

**Ключевые поля:**
- `Server` / `Address`: Показывает, какой DNS-резолвер обрабатывал запрос.
- `Non-authoritative answer`: Указывает, что ответ был получен из кэша резолвера, а не напрямую от авторитарного сервера домена. Это нормальное поведение для большинства запросов.
- `Authoritative answer`: Встречается, если запрос был отправлен непосредственно на авторитарный сервер домена. Это означает, что ответ на 100% актуален.
