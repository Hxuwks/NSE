## Транспортный уровень. TCP vs UDP, порты, различия

**Транспортный уровень** (L4 в модели OSI, Transport layer в TCP/IP) отвечает за связь между **процессами** (приложениями) на разных узлах. Если сетевой уровень (IP) доставляет пакет от одного устройства к другому, то транспортный уровень определяет, **какому именно приложению** на том устройстве эти данные предназначены. Кроме того, транспортный уровень может обеспечивать надёжность, контроль порядка, устранение дубликатов и управление потоком — но не всегда. Всё зависит от протокола.

Главными понятиями здесь являются **порты**, **TCP** и **UDP**.

---

# Порты (1–65535)

**Порт** — это 16-битное число (от 0 до 65535), которое идентифицирует конкретный процесс или сетевую службу на узле. Порты работают на транспортном уровне и позволяют одному устройству одновременно поддерживать множество сетевых соединений с разными приложениями.

Представьте многоквартирный дом:
- **IP-адрес** — это адрес дома (улица, номер).
- **Порт** — это номер квартиры.

Письмо (пакет) приходит на адрес дома, а курьер смотрит на номер квартиры, чтобы отдать письмо именно тому жильцу.

**Диапазоны портов (стандарт IANA):**

| Диапазон | Название | Назначение | Примеры |
|----------|----------|------------|---------|
| 0–1023 | **System ports** (Well-known) | Зарезервированы для системных служб и общеизвестных протоколов. Для использования на этих портах обычно нужны права суперпользователя (root/admin) | 80 (HTTP), 443 (HTTPS), 22 (SSH), 53 (DNS) |
| 1024–49151 | **User ports** (Registered) | Могут использоваться пользовательскими приложениями. IANA может регистрировать конкретные порты за конкретными приложениями | 3306 (MySQL), 5432 (PostgreSQL), 8080 (альтернативный HTTP) |
| 49152–65535 | **Dynamic / Private ports** (Ephemeral) | Временные порты, которые клиентская ОС назначает случайным образом на время соединения. После закрытия соединения порт освобождается | Любой временный порт для исходящего соединения |

**Как это работает на практике:**

Когда ваш браузер открывает сайт `google.com:443`:
1. ОС назначает **исходящий (source) порт** из динамического диапазона (например, 54321).
2. Браузер подключается к **целевому (destination) порту** 443 на сервере Google.
3. Сервер видит: пакет пришёл с порта 54321 на порт 443.
4. Сервер отправляет ответ обратно на порт 54321 вашего компьютера.
5. Ваша ОС по порту 54321 понимает, что данные нужно отдать именно браузеру, а не другому приложению.

# TCP (Transmission Control Protocol)

**TCP** — протокол транспортного уровня с **установлением соединения**, обеспечивающий **гарантированную доставку**, контроль целостности, порядка следования сегментов и управление перегрузками. TCP используется там, где потеря данных недопустима: веб-серфинг (HTTP/HTTPS), электронная почта (SMTP, IMAP), передача файлов (FTP), SSH.

**Ключевые характеристики TCP:**

| Характеристика | Описание |
|----------------|----------|
| Соединение | Ориентирован на соединение. Перед передачей данных устанавливается логический канал (TCP-соединение) через **трёхэтапное рукопожатие** (three-way handshake) |
| Надёжность | Каждый отправленный сегмент должен быть подтверждён (ACK). Если подтверждение не пришло — сегмент отправляется повторно |
| Порядок данных | Данные могут прийти в неправильном порядке из-за маршрутизации. TCP нумерует сегменты (sequence numbers) и переупорядочивает их на получателе |
| Контроль целостности | 16-битная контрольная сумма (checksum) в заголовке TCP. При ошибке сегмент отбрасывается и запрашивается повторно |
| Управление потоком (flow control) | Механизм **окна** (window). Получатель сообщает, сколько данных он готов принять (чтобы отправитель не завалил его) |
| Управление перегрузкой (congestion control) | Отправитель динамически уменьшает скорость при потере пакетов, чтобы не перегружать сеть |
| Передача | Данные передаются как **поток байтов** (byte stream), без сохранения границ сообщений |
| Скорость | Ниже, чем у UDP, из-за накладных расходов на подтверждения, повторные передачи и управление состоянием |

**Структура TCP-сегмента (упрощённо):**

```text
┌─────────────────────────────────────────────────────────────────┐
│                     TCP-ЗАГОЛОВОК (20–60 байт)                  │
├─────────────────┬─────────────────┬─────────────────────────────┤
│ Source port (16)│ Destination port│   Sequence number (32)      │
├─────────────────┴─────────────────┴─────────────────────────────┤
│   Acknowledgment number (32)                                    │
├────────────┬────────┬──────────┬────────┬───────────────────────┤
│ Data offset│Reserved│ Flags    │ Window │ Checksum │ Urgent ptr │
│   (4)      │  (3)   │ (9: FIN, │ (16)   │  (16)    │  (16)     │
│            │        │  SYN,    │        │          │           │
│            │        │  RST,    │        │          │           │
│            │        │  PSH,    │        │          │           │
│            │        │  ACK,    │        │          │           │
│            │        │  URG)    │        │          │           │
├────────────┴────────┴──────────┴────────┴──────────┴───────────┤
│                     Options (0–40 байт)                         │
├─────────────────────────────────────────────────────────────────┤
│                     DATA (полезная нагрузка)                    │
└─────────────────────────────────────────────────────────────────┘
```

**Важные флаги TCP (кратко):**
- `SYN` — установка соединения (synchronize)
- `ACK` — подтверждение получения
- `FIN` — завершение соединения (finish)
- `RST` — сброс соединения (reset, при ошибке)
- `PSH` — немедленная отправка данных без буферизации (push)

**Трёхэтапное рукопожатие (three-way handshake) — установка TCP-соединения:**
```text
Клиент (порт 54321)                Сервер (порт 80)
      |                                    |
      |--------  SYN (seq=x) ------------>|  (1) Клиент: давай соединимся
      |                                    |
      |<------ SYN+ACK (seq=y, ack=x+1) ---|  (2) Сервер: OK, я готов
      |                                    |
      |--------  ACK (ack=y+1) ----------->|  (3) Клиент: принято
      |                                    |
      |<======= ДАННЫЕ (HTTP-запрос) ======>|
```

После установки соединения обе стороны могут отправлять данные. Завершение соединения происходит через обмен `FIN`-пакетами (четыре пакета).

# UDP (User Datagram Protocol)

**UDP** — протокол транспортного уровня **без установления соединения**, который не гарантирует доставку, порядок или целостность данных. UDP используется там, где скорость важнее надёжности, или где приложение само отвечает за контроль: видеозвонки (Zoom, Skype), потоковое видео (YouTube, Twitch), DNS-запросы, онлайн-игры, SNMP, DHCP.

**Ключевые характеристики UDP:**

| Характеристика | Описание |
|----------------|----------|
| Соединение | Без установления соединения (connectionless). Каждая датаграмма отправляется независимо, без предварительного «рукопожатия» |
| Надёжность | Не гарантирует доставку. Пакеты могут теряться, дублироваться или приходить в другом порядке |
| Порядок данных | Не восстанавливает порядок. Если порядок важен — приложение должно его контролировать самостоятельно |
| Контроль целостности | Контрольная сумма (checksum) есть, но при ошибке пакет просто отбрасывается (без повторной отправки) |
| Управление потоком | Отсутствует. Отправитель посылает данные с той скоростью, с которой хочет (может перегрузить получателя) |
| Управление перегрузкой | Отсутствует. UDP может забить канал, не замедляясь (чем иногда пользуются DDoS-атаки) |
| Передача | Данные передаются **датаграммами** (сохраняются границы сообщений). Приложение получает ровно то, что отправил другой конец |
| Скорость | Выше, чем у TCP. Нет задержек на подтверждения, повторные передачи и установку соединения |
| Заголовок | Всего 8 байт (против 20+ у TCP) |

**Структура UDP-датаграммы:**

```text
┌─────────────────────────────────────────────────────────────────┐
│                     UDP-ЗАГОЛОВОК (8 байт)                      │
├─────────────────┬─────────────────┬─────────────────────────────┤
│ Source port (16)│ Destination port│   Length (16)               │
├─────────────────┴─────────────────┴─────────────────────────────┤
│   Checksum (16)                                                │
├─────────────────────────────────────────────────────────────────┤
│                     DATA (полезная нагрузка)                    │
└─────────────────────────────────────────────────────────────────┘
```

**Особенность UDP:** не хранит состояние соединений, поэтому может обслуживать значительно больше клиентов при меньших затратах памяти и процессора.

# TCP vs UDP — полное сравнение

| Характеристика | TCP | UDP |
|----------------|-----|-----|
| **Название** | Transmission Control Protocol | User Datagram Protocol |
| **Соединение** | Ориентирован на соединение (connection-oriented) | Без соединения (connectionless) |
| **Рукопожатие** | Да (SYN, SYN-ACK, ACK) | Нет |
| **Надёжность** | Гарантированная доставка (подтверждения + повторы) | Не гарантирует доставку |
| **Порядок данных** | Сохраняет порядок (sequence numbers) | Не сохраняет порядок |
| **Потеря пакетов** | Повторная отправка | Без повторной отправки (пакет теряется) |
| **Дубликаты** | Устраняет | Могут быть дубликаты |
| **Управление потоком** | Да (window size) | Нет |
| **Управление перегрузкой** | Да (сложные алгоритмы: Reno, Cubic и др.) | Нет (может забить сеть) |
| **Размер заголовка** | 20–60 байт | 8 байт |
| **Скорость** | Ниже (из-за накладных расходов) | Выше (минимальные задержки) |
| **Границы сообщений** | Поток байтов (границы не сохраняются) | Датаграммы (границы сохраняются) |
| **Типичное применение** | HTTP, HTTPS, SSH, FTP, SMTP, Telnet | DNS, DHCP, VoIP, видео, онлайн-игры, SNMP |

**ASCII-схема: выбор между TCP и UDP:**

```text
                        ┌─────────────────────────┐
                        │   Какие требования?     │
                        └───────────┬─────────────┘
                                    │
            ┌───────────────────────┴───────────────────────┐
            │                                               │
            ▼                                               ▼
┌───────────────────────┐                       ┌───────────────────────┐
│   Данные нельзя терять │                       │ Скорость критичнее    │
│   Порядок важен        │                       │ Потери допустимы      │
│   Надёжность > скорости│                       │ 1 потерянный пакет     │
│                        │                       │ лучше, чем задержка    │
└───────────┬───────────┘                       └───────────┬───────────┘
            │                                               │
            ▼                                               ▼
    ┌───────────────┐                               ┌───────────────┐
    │      TCP      │                               │      UDP      │
    │  (надёжность) │                               │   (скорость)  │
    └───────────────┘                               └───────────────┘
           │                                               │
           ▼                                               ▼
    HTTP, HTTPS, SSH,                              Видео, аудио,
    FTP, почта, Telegram                          онлайн-игры, DNS
```

# Инкапсуляция транспортного уровня (пример с TCP)

Важный момент: данные транспортного уровня (TCP-сегмент или UDP-датаграмма) становятся **полезной нагрузкой** для сетевого уровня (IP-пакета).

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                           ПРИКЛАДНЫЕ ДАННЫЕ                                 │
│                      "GET /index.html HTTP/1.1"                             │
└─────────────────────────────────────────────────────────────────────────────┘
                                      │
                                      ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│  ТРАНСПОРТНЫЙ УРОВЕНЬ (TCP или UDP)                                        │
│  ┌─────────────────────────────────────────────────────────────────────┐    │
│  │ TCP-заголовок (20–60 байт)          │  Данные (HTTP-запрос)         │    │
│  │ ┌──────────┬──────────┬──────────┐  │                               │    │
│  │ │src port: │dst port: │ seq num: │  │ "GET /index.html HTTP/1.1"    │    │
│  │ │  54321   │   80     │  12345   │  │ Host: google.com ...          │    │
│  │ │ ack num: │ flags    │ window   │  │                               │    │
│  │ │  12346   │ SYN,ACK  │  8192    │  │                               │    │
│  │ └──────────┴──────────┴──────────┘  │                               │    │
│  └─────────────────────────────────────────────────────────────────────┘    │
│                         ВЕСЬ БЛОК = TCP-СЕГМЕНТ (или UDP-датаграмма)        │
└─────────────────────────────────────────────────────────────────────────────┘
                                      │
                                      ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│  СЕТЕВОЙ УРОВЕНЬ (IP)                                                       │
│  ┌─────────────────────────────────────────────────────────────────────┐    │
│  │ IP-заголовок (20–60 байт)            │  TCP-сегмент (payload)       │    │
│  │ ┌──────────┬──────────┬──────────┐   │  ┌──────────────────────┐     │    │
│  │ │src IP:   │dst IP:   │Protocol: │   │  │ TCP-заголовок │ HTTP │     │    │
│  │ │192.168.1.10│8.8.8.8  │   6 (TCP)│   │  └──────────────────────┘     │    │
│  │ └──────────┴──────────┴──────────┘   │                               │    │
│  └─────────────────────────────────────────────────────────────────────┘    │
│                         ВЕСЬ БЛОК = IP-ПАКЕТ                               │
└─────────────────────────────────────────────────────────────────────────────┘
```

> **Обратите внимание:** в IP-заголовке есть поле **Protocol**, которое указывает, какой протокол транспортного уровня находится внутри: 6 для TCP, 17 для UDP.

# И ещё кое-что: SCTP 

Существует также третий протокол транспортного уровня — **SCTP** (Stream Control Transmission Protocol). Он сочетает особенности TCP (надёжность, порядок) и UDP (сохранение границ сообщений, мультиплексирование нескольких потоков). Используется в основном в телеком-индустрии (сигнализация SS7, LTE). В обычных сетях встречается редко.
