**DHCP** (Dynamic Host Configuration Protocol) — это сетевой протокол прикладного уровня (L7 в модели OSI), позволяющий сетевым устройствам (клиентам) автоматически получать необходимые параметры для работы в IP-сети. DHCP является развитием более старого протокола BOOTP (Bootstrap Protocol) и определён в RFC 2131 (1997 год).

Основная задача DHCP — **централизованное и динамическое управление** сетевыми настройками, что избавляет администратора от необходимости вручную настраивать каждое устройство в сети. Вместо этого, клиент при подключении обращается к DHCP-серверу и получает от него весь необходимый набор параметров.

---

### 1. Назначение и фундаментальные принципы

В отличие от статической конфигурации, где IP-адрес, маска подсети, шлюз и DNS-серверы вводятся вручную для каждого устройства, DHCP предоставляет следующие возможности:

- **Автоматическое назначение IP-адресов:** Сервер выдаёт свободный адрес из заранее определённого пула (диапазона).
- **Централизованное управление:** Все сетевые параметры (адрес шлюза, DNS-серверы, маршруты, имя домена и др.) хранятся в одном месте — на сервере.
- **Динамическое переиспользование адресов:** Адреса выдаются во временное пользование (аренда, lease). После истечения срока аренды адрес возвращается в пул и может быть выдан другому устройству. Это критически важно для сетей с большим количеством мобильных устройств.
- **Предотвращение конфликтов адресов:** Сервер отслеживает, какие адреса уже выданы, и не назначает их повторно.

**Ключевые характеристики:**

- **Работает поверх UDP:** Клиент отправляет запросы с порта 68 на порт 67 сервера.
- **Модель «клиент-сервер»:** DHCP-сервер отвечает на запросы клиентов, предоставляя им конфигурационные параметры.
- **Поддержка BOOTP:** DHCP обратно совместим с BOOTP (устаревший протокол для загрузки бездисковых станций), что позволяет обслуживать оба типа клиентов.
- **Типы IP-адресации, поддерживаемые DHCP:**

| Тип | Описание |
| :--- | :--- |
| **Динамическая (Dynamic)** | Адрес выдаётся на ограниченный срок (аренду). После истечения аренды адрес может быть выдан другому устройству. Это основной режим работы. |
| **Автоматическая (Automatic)** | Адрес выдаётся навсегда (без срока аренды). Сервер просто назначает статический адрес из пула, но делает это автоматически. |
| **Статическая (Manual / Reservation)** | Адрес закреплён за конкретным устройством по его MAC-адресу. Сервер всегда выдаёт один и тот же IP-адрес данному клиенту. |

### 2. Структура DHCP-сообщения

DHCP-сообщения инкапсулируются внутри UDP-датаграмм. Структура сообщения (упрощённо) выглядит следующим образом:

| Поле | Размер (байт) | Назначение |
| :--- | :--- | :--- |
| **op** | 1 | Тип сообщения: 1 = запрос клиента, 2 = ответ сервера |
| **htype** | 1 | Тип аппаратного адреса (для Ethernet — 1) |
| **hlen** | 1 | Длина аппаратного адреса в байтах (для MAC-48 — 6) |
| **hops** | 1 | Количество ретрансляций (используется для прохождения через DHCP-реле) |
| **xid** | 4 | Уникальный идентификатор транзакции (клиент генерирует для связывания запроса и ответа) |
| **secs** | 2 | Время (в секундах) с момента начала процесса получения адреса |
| **flags** | 2 | Флаги (например, широковещательный флаг) |
| **ciaddr** | 4 | IP-адрес клиента (если уже есть) |
| **yiaddr** | 4 | Предлагаемый сервером IP-адрес для клиента |
| **siaddr** | 4 | IP-адрес следующего сервера (для BOOTP) |
| **giaddr** | 4 | IP-адрес DHCP-реле (если запрос пришёл через релейный агент) |
| **chaddr** | 16 | MAC-адрес клиента (аппаратный адрес) |
| **sname** | 64 | Имя сервера (опционально) |
| **file** | 128 | Имя загрузочного файла (для BOOTP) |
| **options** | Переменная | Поля переменной длины, содержащие основную информацию: тип сообщения, предлагаемый IP, маска подсети, шлюз, DNS-серверы, время аренды и т.д. |

### 3. Процесс получения IP-адреса (4-шаговое рукопожатие, DORA)

Процесс получения динамического IP-адреса называется **DORA**, по первым буквам четырёх этапов:

1.  **DISCOVER** (Обнаружение): Клиент ищет DHCP-сервер.
2.  **OFFER** (Предложение): Сервер предлагает клиенту IP-адрес.
3.  **REQUEST** (Запрос): Клиент запрашивает предложенный адрес.
4.  **ACKNOWLEDGE** (Подтверждение): Сервер подтверждает выдачу адреса.

**Пошаговое описание:**

```text
Клиент (MAC: AA:AA:AA:AA:AA:AA)           DHCP-сервер (192.168.1.1)
    │                                                │
    │  1. DHCPDISCOVER (широковещательно)             │
    │  src: 0.0.0.0:68, dst: 255.255.255.255:67      │
    │  "Я новый клиент (MAC: ...). Кто-нибудь есть?"  │
    │────────────────────────────────────────────────>│
    │                                                │
    │  2. DHCPOFFER                                   │
    │  "Привет! Я могу предложить тебе 192.168.1.100, │
    │   маску 255.255.255.0, шлюз 192.168.1.1,       │
    │   DNS 8.8.8.8, аренда на 24 часа."              │
    │<────────────────────────────────────────────────│
    │                                                │
    │  3. DHCPREQUEST (широковещательно)              │
    │  "Я выбираю адрес 192.168.1.100."              │
    │  (Широковещательно, чтобы сообщить остальным   │
    │   серверам, что их предложения отклонены)       │
    │────────────────────────────────────────────────>│
    │                                                │
    │  4. DHCPACK                                     │
    │  "OK, адрес 192.168.1.100 закреплён за тобой.  │
    │   Вот твоя конфигурация."                       │
    │<────────────────────────────────────────────────│
```

**Подробное описание каждого этапа:**

1.  **DHCPDISCOVER**:
    - Клиент, не имея IP-адреса, отправляет **широковещательный** UDP-пакет (dst = `255.255.255.255`, src = `0.0.0.0`) на порт 67.
    - В сообщении указывается свой MAC-адрес (chaddr) и генерируется уникальный идентификатор транзакции (xid).
    - Цель: найти всех DHCP-серверов в локальном широковещательном домене.

2.  **DHCPOFFER**:
    - DHCP-сервер, получив DISCOVER, резервирует свободный IP-адрес из своего пула и отправляет клиенту **направленный** (или широковещательный, если клиент не имеет IP) ответ.
    - В ответе содержатся: предлагаемый IP-адрес (yiaddr), маска подсети, шлюз, DNS-серверы, время аренды и другие параметры.
    - Если в сети несколько DHCP-серверов, каждый из них может отправить своё предложение. Клиент выберет одно из них.

3.  **DHCPREQUEST**:
    - Клиент выбирает одно из предложений (обычно первое) и отправляет **широковещательный** запрос, в котором явно указывает выбранный IP-адрес и сервер, от которого он принял предложение.
    - Широковещательная отправка необходима для того, чтобы **информировать остальные DHCP-серверы** о том, что их предложения были отклонены. Те серверы, чьи предложения не были выбраны, возвращают зарезервированные адреса обратно в пул свободных.

4.  **DHCPACK**:
    - Выбранный DHCP-сервер получает REQUEST, фиксирует, что адрес официально выдан клиенту, и отправляет клиенту подтверждение (**DHCPACK**) с окончательными параметрами конфигурации (маска, шлюз, DNS, TTL и т.д.).
    - Клиент получает ACK, настраивает свой сетевой интерфейс с полученными параметрами и начинает использовать адрес.
    - **Важно:** В случае, если сервер не может подтвердить выдачу (например, кто-то уже занял этот адрес), он отправляет **DHCPNAK** (Negative Acknowledgment). Клиент, получив NAK, должен немедленно начать процесс заново, начиная с DISCOVER.

### 4. Аренда (Lease) и её продление

Адрес выдаётся клиенту **на ограниченный срок** — время аренды (lease time). Это время обычно настраивается администратором и может варьироваться от нескольких минут (в сетях с высокой мобильностью, например, Wi-Fi в кафе) до нескольких дней или недель.

**Жизненный цикл аренды:**

- **T1 (время обновления, renewal):** Обычно это **50%** от времени аренды. По достижении этого времени клиент пытается **продлить** аренду напрямую у того же сервера, который её выдал. Он отправляет **DHCPREQUEST** (уже не широковещательно, а направленно) на сервер. Если сервер отвечает **DHCPACK**, аренда продлевается.
- **T2 (время повторной привязки, rebinding):** Обычно это **87.5%** от времени аренды. Если клиент не получил ответа от своего сервера на T1 (сервер недоступен), он переходит в режим **rebinding** и отправляет **широковещательный** DHCPREQUEST. Теперь любой DHCP-сервер в сети может ответить и продлить аренду.
- **Истечение срока:** Если клиент не получил ответ ни на T1, ни на T2, по истечении полного срока аренды он **обязан** прекратить использование IP-адреса и начать процесс DORA с нуля (с DISCOVER).

**ASCII-схема жизненного цикла аренды:**

```text
Время:  ────┬─────────────────────────┬─────────────────────────┬───────────────────────►
            │         0               │         50% (T1)        │      87.5% (T2)        │
            ▼                         ▼                         ▼                         ▼
        ┌─────────┐               ┌─────────┐               ┌─────────┐             ┌──────────┐
        │ АРЕНДА  │──────────────▶│ ПРОДЛЕ- │──────────────▶│ ПОВТОР- │────────────▶│ ИСТЕЧЕ-  │
        │ НАЧАЛА  │               │ НИЕ     │               │ НАЯ     │             │ НИЕ      │
        │ (ACK)   │               │ (Renew) │               │ ПРИВЯЗ- │             │ (Expired)│
        │         │               │         │               │ КА      │             │          │
        │         │               │         │               │ (Rebind)│             │          │
        └─────────┘               └─────────┘               └─────────┘             └──────────┘
          ▲                           │                         │                         │
          │                           ▼                         ▼                         ▼
          │                    DHCPREQUEST               DHCPREQUEST               DHCPDISCOVER
          │                    направленный              широковещательный         (с нуля)
          │                    на свой сервер            на любой сервер
          │                           │                         │
          └───────────────────────────┴─────────────────────────┘
                                    Если ответ (ACK) получен, таймер аренды сбрасывается
```

### 5. Роль DHCP в сетях и инфраструктуре

- **DHCP-сервер:** Обычно реализован как служба на выделенном сервере (в Windows Server — роль DHCP, в Linux — пакет `isc-dhcp-server` или `dnsmasq`) или встроен в маршрутизаторы/коммутаторы (домашние роутеры).
- **DHCP-клиент:** Присутствует в любой современной ОС (Windows, Linux, macOS, Android, iOS) и по умолчанию настроен на получение IP-адреса автоматически.
- **DHCP-реле (DHCP Relay Agent):** Важный компонент для больших сетей. Так как DISCOVER отправляется широковещательно, он не проходит через маршрутизаторы. Если DHCP-сервер находится в другой подсети, необходимо настроить **DHCP-реле** на маршрутизаторе. Релейный агент перехватывает широковещательные запросы от клиентов, упаковывает их в обычный UDP-пакет и направляет их непосредственно на DHCP-сервер (указывая в поле `giaddr` свой IP-адрес, чтобы сервер знал, из какой подсети пришёл запрос и из какого пула выдавать адрес).

### 6. Безопасность и уязвимости DHCP

DHCP, как и ARP, не содержит механизмов аутентификации, что делает его уязвимым:

| Атака | Описание | Последствия |
| :--- | :--- | :--- |
| **Rogue DHCP Server (Поддельный DHCP-сервер)** | Злоумышленник подключает в сеть свой DHCP-сервер, который отвечает на DISCOVER быстрее легитимного. | Клиенты получают от злоумышленника неправильные настройки (шлюз, DNS), что позволяет осуществлять атаки Man-in-the-Middle. |
| **DHCP Starvation (Голодание)** | Злоумышленник отправляет огромное количество DISCOVER-запросов с поддельными MAC-адресами. | Сервер исчерпывает свой пул адресов, и новые легитимные клиенты не могут получить IP-адрес (отказ в обслуживании, DoS). |

**Методы защиты:**

- **DHCP Snooping** — технология на управляемых коммутаторах, которая назначает доверенные порты (к которым подключены легитимные DHCP-серверы) и блокирует все DHCP-сообщения от нетрустных портов.
- Использование **802.1X** для аутентификации устройств на канальном уровне.
- Ручная настройка статических адресов для критически важных узлов (серверы, шлюзы).

### 7. Связь с другими уровнями (кратко)

```text
┌─────────────────────────────────────────────────────────────────┐
│ ПРИКЛАДНОЙ УРОВЕНЬ (L7)                                       │
│   DHCP (порт 67 - сервер, порт 68 - клиент)                   │
└─────────────────────────────┬───────────────────────────────────┘
                              │ DHCP-сообщение (UDP-датаграмма)
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│ ТРАНСПОРТНЫЙ УРОВЕНЬ (L4)                                     │
│   UDP (без соединения, быстрая доставка)                       │
└─────────────────────────────┬───────────────────────────────────┘
                              │ UDP-датаграмма
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│ СЕТЕВОЙ УРОВЕНЬ (L3)                                           │
│   IP-пакет (src 0.0.0.0, dst 255.255.255.255 для DISCOVER)    │
└─────────────────────────────┬───────────────────────────────────┘
                              │ IP-пакет
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│ КАНАЛЬНЫЙ УРОВЕНЬ (L2)                                         │
│   Ethernet-кадр с MAC-адресами (broadcast для DISCOVER)       │
└─────────────────────────────────────────────────────────────────┘
```
