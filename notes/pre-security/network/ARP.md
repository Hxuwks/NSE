# Протокол разрешения адресов (ARP)

**Протокол разрешения адресов** (англ. *Address Resolution Protocol*, **ARP**) — протокол канального (в терминах модели OSI) или сетевого (в терминах модели TCP/IP, где он часто приписывается к интерфейсу между L2 и L3) уровня, предназначенный для преобразования сетевых адресов (IPv4) в аппаратные адреса (MAC-адреса) в локальных сетях. ARP определён в RFC 826 (1982 год) и является фундаментальным элементом стеков протоколов IPv4 в широковещательных сетях, таких как Ethernet и Wi-Fi.

> **Примечание:** в протоколе IPv6 аналогичный функционал реализован протоколом **NDP** (Neighbor Discovery Protocol), а не ARP.

---

## 1. Назначение и область применения

В сетях стандарта IEEE 802 (Ethernet, Wi-Fi, Token Ring и др.) адресация на канальном уровне осуществляется по **MAC-адресам** — 48-битным идентификаторам, жёстко привязанным к сетевым интерфейсам. Однако прикладные процессы и пользователи оперируют **IP-адресами** (32 бит для IPv4), которые обладают иерархической структурой и могут динамически изменяться.

Возникает задача: как, зная IP-адрес целевого узла в пределах одного широковещательного домена (одной подсети), определить его MAC-адрес, чтобы сформировать кадр канального уровня? Эту задачу и решает ARP.

**Основная функция ARP** — динамическое отображение («связывание») IPv4-адреса и MAC-адреса в локальной сети.

---

## 2. Принцип работы (базовый сценарий)

Работа ARP основана на широковещательных запросах и индивидуальных ответах. Рассмотрим случай, когда узел A (отправитель) хочет передать данные узлу B (получатель) в пределах одной подсети, но не знает MAC-адрес узла B.

### 2.1. ARP-запрос (ARP request)

1. Узел A формирует **ARP-запрос** — специальный кадр канального уровня, в котором указаны:
   - IP-адрес узла A (источник),
   - MAC-адрес узла A (источник),
   - IP-адрес узла B (целевой, который требуется разрешить),
   - MAC-адрес цели в поле «адрес отправителя» (заполняется нулями, так как он неизвестен).

2. В качестве **целевого MAC-адреса** кадра указывается широковещательный адрес `FF:FF:FF:FF:FF:FF`. Это гарантирует, что кадр будет принят **всеми** узлами в данном широковещательном домене (сегменте локальной сети).

3. Кадр отправляется в сеть.

### 2.2. Обработка запроса узлами

- Каждый активный узел в сети принимает широковещательный кадр и передаёт его на уровень ARP в стеке протоколов.
- Узел сравнивает поле `target IP-адрес` (целевой IP) со своим собственным IP-адресом.
- **Если адреса не совпадают** — узел молча игнорирует запрос (никакого ответа не отправляется).
- **Если адреса совпадают** — узел B идентифицирует себя как целевой.

### 2.3. ARP-ответ (ARP reply)

Узел B формирует **ARP-ответ**:
- Заполняет поля: свой MAC-адрес (как ответчик), свой IP-адрес, а также скопированные из запроса IP и MAC отправителя A.
- Отправляет этот ответ **непосредственно узлу A** (указывая в кадре MAC-адрес A в качестве целевого, а не широковещательный адрес).

Узел A получает ARP-ответ, извлекает из него MAC-адрес узла B и сохраняет полученную пару (IP → MAC) в локальном **ARP-кэше**.

### 2.4. Обмен данными

После получения MAC-адреса узла B, узел A может формировать обычные IP-кадры (например, с данными приложения), указывая в заголовке канального уровня MAC-адрес B как целевой.

---

## 3. Формат ARP-сообщения

ARP-сообщения инкапсулируются непосредственно внутри кадров канального уровня (например, Ethernet). Для Ethernet поле `EtherType` устанавливается в значение `0x0806`, сигнализирующее, что кадр содержит ARP-пакет, а не IP-датаграмму.

Ниже приведена структура ARP-сообщения (RFC 826):

| Поле | Размер (байт) | Назначение |
|------|---------------|-------------|
| **Hardware Type (HTYPE)** | 2 | Тип аппаратного адреса. Для Ethernet = 1 |
| **Protocol Type (PTYPE)** | 2 | Тип протокольного (сетевого) адреса. Для IPv4 = 0x0800 |
| **Hardware Address Length (HLEN)** | 1 | Длина аппаратного адреса в байтах. Для MAC-48 = 6 |
| **Protocol Address Length (PLEN)** | 1 | Длина протокольного адреса в байтах. Для IPv4 = 4 |
| **Operation (OPER)** | 2 | Тип операции: 1 = ARP-запрос, 2 = ARP-ответ |
| **Sender Hardware Address (SHA)** | HLEN (6) | MAC-адрес отправителя сообщения |
| **Sender Protocol Address (SPA)** | PLEN (4) | IP-адрес отправителя |
| **Target Hardware Address (THA)** | HLEN (6) | MAC-адрес получателя (в запросе — нули) |
| **Target Protocol Address (TPA)** | PLEN (4) | IP-адрес получателя |

**Общая длина ARP-сообщения для Ethernet/IPv4:** 8 байт (первые 8 полей фиксированной структуры) + 2×6 + 2×4 = **28 байт**.

## 4. ARP-кэш

Для повышения эффективности ARP реализует **кэширование**. Каждый узел поддерживает таблицу (ARP-кэш), в которой хранятся недавно разрешённые соответствия «IP-адрес → MAC-адрес».

### 4.1. Характеристики ARP-кэша

- **Время жизни записи (таймер)**: типичные значения от 30 секунд до 20 минут (зависит от реализации ОС).
- **Статические записи**: администратор может вручную добавить постоянные записи, не подверженные старению (например, для важных шлюзов).
- **Автоматическое обновление**: при успешном использовании существующей записи её таймер может сбрасываться (soft state).

### 4.2. Просмотр и управление ARP-кэшем

**Linux / macOS / Unix:**
```bash
arp -n                    # показать ARP-кэш (без разрешения имён)
ip neigh                  # современная команда (iproute2)
ip neigh del 192.168.1.1 dev eth0   # удалить запись
```

**Windows:**
```cmd
arp -a                    # показать ARP-кэш
arp -d                    # очистить кэш
```

## 5. Варианты и расширения ARP

### 5.1. Gratuitous ARP (произвольный ARP)

**Gratuitous ARP** — ARP-пакет (обычно запрос, иногда ответ), отправляемый узлом **без предварительного запроса** от другого узла. Основные применения:

1. **Обновление ARP-кэша других узлов** при изменении MAC-адреса узла (например, при смене сетевой карты или в кластерах виртуальных IP).
2. **Обнаружение конфликта IP-адресов** (Duplicate Address Detection, DAD). Узел отправляет ARP-запрос на свой собственный IP-адрес и ожидает ответ. Если ответ пришёл — адрес уже используется.
3. **Инициализация сетевого интерфейса** (прямая широковещательная рассылка для ускорения работы коммутаторов).

### 5.2. Proxy ARP (ARP-посредник / прокси-ARP)

**Proxy ARP** — механизм, при котором маршрутизатор отвечает на ARP-запросы, предназначенные для узлов, находящихся в **других** подсетях. Маршрутизатор «притворяется» целевым узлом, подставляя свой MAC-адрес.

Сценарий:
- Узел A отправляет ARP-запрос на IP-адрес B, который находится в другой сети.
- Маршрутизатор R (настроенный как Proxy ARP) отвечает своим MAC-адресом.
- Узел A отправляет все данные на R, а R осуществляет IP-маршрутизацию к B.

**Когда используется:** переходные сети без явной настройки шлюза по умолчанию, но с включённым прокси-ARP на маршрутизаторе. Сегодня считается устаревшим методом (исторически использовался для объединения подсетей в сетях типа «ленивая маршрутизация»).

## 6. ARP в разных сетевых средах

| Тип сети | Особенности ARP |
|----------|----------------|
| **Ethernet (широковещательный)** | Базовая модель: ARP работает в полной мере, используются широковещательные кадры `FF:FF:FF:FF:FF:FF`. |
| **Точка-точка (PPP, SLIP)** | ARP не требуется, так как в таких линках известен единственный сосед. |
| **Wi-Fi (802.11)** | Полностью аналогично Ethernet; ARP-кадры передаются как любые другие кадры (широковещательные рассылки могут иметь ограничения в некоторых конфигурациях). |
| **VLAN (802.1Q)** | ARP работает внутри каждого VLAN как в отдельной широковещательной сети. |
| **IPv6** | ARP не используется, вместо него — **Neighbor Discovery Protocol (NDP)** на основе ICMPv6. |

---

## 7. Проблемы и уязвимости ARP

### 7.1. ARP-спуфинг (ARP spoofing, ARP poisoning)

Поскольку протокол ARP не содержит механизмов аутентификации, любой узел в широковещательном домене может отправлять поддельные ARP-ответы. Это позволяет злоумышленнику:

- **Подменять соответствие** IP-адрес → MAC-адрес в кэшах других узлов.
- **Перенаправлять трафик** на свой компьютер (атака «человек посередине», Man-in-the-Middle).
- **Вызывать отказ в обслуживании** (DoS), подставляя несуществующие MAC-адреса или широковещательные ответы.

**Методы защиты:**
- Статические ARP-записи (для критических узлов, например шлюзов).
- Динамическое обнаружение ARP-атак (DAI, Dynamic ARP Inspection) на управляемых коммутаторах.
- Сегментация сети (разделение на малые широковещательные домены с помощью VLAN).
- Использование сетевых протоколов с шифрованием (IPsec, SSH, HTTPS) — снижает ущерб от перехвата.

### 7.2. Широковещательный шторм (broadcast storm)

Злоумышленник или неисправное сетевое оборудование могут генерировать огромное количество широковещательных ARP-запросов, перегружая сеть. Проблема решается настройкой ограничений (broadcast throttling) и сегментацией.

## 8. Пример работы ARP (трассировка пакетов)

Ниже приведён пример обмена ARP-сообщениями в сети Ethernet между узлами A (192.168.1.10, MAC AA:AA:AA:AA:AA:AA) и B (192.168.1.20, MAC BB:BB:BB:BB:BB:BB).

**Шаг 1. ARP-запрос от A (широковещательный):**
```text
Ethernet-кадр:
  dest MAC = FF:FF:FF:FF:FF:FF
  src MAC  = AA:AA:AA:AA:AA:AA
  type     = 0x0806

ARP-сообщение:
  OPER = 1 (запрос)
  SHA = AA:AA:AA:AA:AA:AA
  SPA = 192.168.1.10
  THA = 00:00:00:00:00:00
  TPA = 192.168.1.20
```

**Шаг 2. ARP-ответ от B (напрямую узлу A):**
```text
Ethernet-кадр:
  dest MAC = AA:AA:AA:AA:AA:AA
  src MAC  = BB:BB:BB:BB:BB:BB
  type     = 0x0806

ARP-сообщение:
  OPER = 2 (ответ)
  SHA = BB:BB:BB:BB:BB:BB
  SPA = 192.168.1.20
  THA = AA:AA:AA:AA:AA:AA
  TPA = 192.168.1.10
```

**Результат:** узел A добавляет в ARP-кэш запись `192.168.1.20 -> BB:BB:BB:BB:BB:BB`.

## 10. Резюме (основные положения)

1. **ARP** обеспечивает динамическое отображение IPv4-адресов в MAC-адреса в широковещательных локальных сетях (прежде всего Ethernet).
2. Протокол работает по схеме «запрос-ответ»: запрос отправляется широковещательно, ответ — индивидуально.
3. Полученные соответствия сохраняются в **ARP-кэше** на ограниченное время (от секунд до минут).
4. ARP уязвим к атакам **подмены (спуфинга)** из-за отсутствия аутентификации.
5. В IPv6 ARP заменён протоколом **Neighbor Discovery (NDP)**.
6. Существуют расширения: **Gratuitous ARP** (обнаружение конфликтов, обновление кэшей) и **Proxy ARP** (маршрутизатор отвечает за удалённые узлы).
