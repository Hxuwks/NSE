**Сетевой уровень** (или Internet layer в модели TCP/IP) отвечает за передачу данных между разными сетями, в отличие от канального уровня, который работает только внутри одной локальной сети. Если канальный уровень доставляет фрейм от одного MAC-адреса к другому в пределах одного broadcast-домена, то сетевой уровень позволяет пересылать пакеты через множество маршрутизаторов, целые континенты. Главными понятиями здесь являются IP-адрес, маска подсети, таблица маршрутизации и шлюз (gateway).

# IP-адрес

**IP-адрес** (Internet Protocol address) — уникальный 32-битный идентификатор сетевого интерфейса в сети IPv4 (существует также 128-битный IPv6, но для базового понимания начнём с IPv4). В отличие от MAC-адреса, который «прошит» производителем и не меняется, IP-адрес назначается программно (вручную администратором или автоматически через DHCP). Именно IP-адрес используется для сквозной адресации: он остаётся неизменным на всём пути от отправителя до получателя, в то время как MAC-адреса меняются при каждом перескоке (hop) через маршрутизатор.

Для удобства чтения 32-битный IP-адрес разделяют на 4 октета (байта) и записывают в десятичном виде через точку. Каждое число — от 0 до 255.

**Пример:** `192.168.1.10`

**Внутреннее представление (биты):**
```
192.168.1.10  →  11000000.10101000.00000001.00001010
```

# Маска подсети

**Маска подсети** — это 32-битное число, которое показывает, какая часть IP-адреса относится к адресу **сети** (network), а какая — к адресу **узла** (host). Без маски IP-адрес сам по себе бесполезен — непонятно, где заканчивается сеть и начинается конкретное устройство.

В маске биты, равные **1**, обозначают часть сети. Биты, равные **0**, — часть узла.

**Пример маски:** `255.255.255.0`  
В битах: `11111111.11111111.11111111.00000000`

**Как вычислить адрес сети (логическое И между IP и маской):**
```
IP-адрес:     192.168.1.10  →  11000000.10101000.00000001.00001010
Маска:        255.255.255.0 →  11111111.11111111.11111111.00000000
                          AND (побитовое умножение)
Адрес сети:   192.168.1.0   →  11000000.10101000.00000001.00000000
```

**Адрес широковещания (broadcast)** для этой сети — все биты узла становятся единицами:
```
Broadcast:    192.168.1.255 →  11000000.10101000.00000001.11111111
```

# Префиксная запись (/24, /16, /8)

Вместо громоздкой записи маски четырьмя числами используют **длину префикса** — количество единичных битов в маске. Это называется CIDR (Classless Inter-Domain Routing).

| Запись | Маска (десятично) | Кол-во узлов | Пример сети | Где используется |
|--------|-------------------|--------------|-------------|-------------------|
| **/8**  | `255.0.0.0`       | 16 777 214   | `10.0.0.0`  | Крупные сети, приватные диапазоны |
| **/16** | `255.255.0.0`     | 65 534       | `172.16.0.0` | Средние сети (датацентры, офисы) |
| **/24** | `255.255.255.0`   | 254          | `192.168.1.0` | Домашние и офисные сети (самый популярный) |
| **/30** | `255.255.255.252` | 2            | `10.0.0.0`  | Точка-точка между маршрутизаторами |
| **/32** | `255.255.255.255` | 1            | конкретный хост | Обозначение одного устройства |
| **/0**  | `0.0.0.0`         | вся сеть IPv4 | `0.0.0.0`   | Маршрут по умолчанию (default gateway) |

**Формула числа узлов:** `2^(32 − префикс) − 2`  
Минус 2, потому что один адрес зарезервирован под **адрес сети** (все биты узла в 0), а другой — под **broadcast** (все биты узла в 1).

**Пример для `/24`:** `2^(8) − 2 = 256 − 2 = 254` устройства.

**Пример для `/30`** (часто используется для линков между маршрутизаторами): всего 4 адреса, из них полезных — 2.

# CIDR и бесклассовая адресация

Исторически существовала **классовая адресация** (классы A, B, C с фиксированными масками /8, /16, /24), но она была слишком негибкой. Например, компании с 300 компьютерами приходилось выделять целый класс B (/16) на 65 тысяч адресов — огромная трата.

Сейчас везде используется **бесклассовая адресация (CIDR)**. Маска может быть любой: `/21`, `/27`, `/19`. Это позволяет нарезать адресное пространство кусками нужного размера — как пиццу на куски разного диаметра.

**Пример CIDR:**  
Сеть `192.168.8.0/21` (маска `255.255.248.0`) охватывает адреса с `192.168.8.0` по `192.168.15.255`. Это 8 последовательных сетей /24, объединённых в одну.

# Специальные IP-адреса

Некоторые адреса имеют особое назначение и не могут быть назначены обычным устройствам:

| Адрес | Назначение | Пояснение |
|-------|------------|-----------|
| `0.0.0.0` | «Любой адрес» | Используется, когда устройство ещё не знает свой IP (например, при запросе DHCP) или как маршрут по умолчанию |
| `127.0.0.1` | localhost | Петлевой интерфейс (loopback). Пакет никуда не уходит, сразу возвращается обратно — для тестов |
| `255.255.255.255` | Ограниченный broadcast | Отправляется всем устройствам в текущей локальной сети, даже если они не знают свою сеть |
| `192.168.0.0/16` | Приватные адреса | Не маршрутизируются в интернете. Используются в локальных сетях (офисы, дома) |
| `10.0.0.0/8` | Приватные адреса | Крупные приватные сети |
| `172.16.0.0/12` | Приватные адреса | Средние приватные сети |
| `169.254.0.0/16` | Link-local (APIPA) | Автоматически назначается, когда нет DHCP. Работает только внутри одной подсети |

# Маршрутизация (как пакеты путешествуют между сетями)

**Маршрутизация** — процесс передачи IP-пакета от отправителя к получателю, когда они находятся в **разных сетях**. Если отправитель и получатель в одной сети (их адреса сети совпадают), пакет идёт напрямую — маршрутизатор не нужен. Если в разных — нужен **шлюз** (gateway).

Ключевой принцип работы маршрутизатора: он смотрит только на **IP-адрес назначения** в пакете и свою **таблицу маршрутизации**, после чего принимает решение:
> «Через какой интерфейс и какому следующему маршрутизатору (next hop) отправить пакет?»

## Таблица маршрутизации (пример)

Упрощённая таблица маршрутизации для домашнего роутера:

| Сеть назначения | Префикс | Шлюз (Gateway) | Интерфейс |
|----------------|---------|----------------|------------|
| `192.168.1.0`  | `/24`   | напрямую (0.0.0.0) | LAN (eth0) |
| `0.0.0.0`      | `/0`    | `95.64.12.1`   | WAN (eth1) |

**Правила выбора маршрута:**
1. **Наиболее точное совпадение** (самый длинный префикс). Если пакет идёт на `192.168.1.100`, сработает правило `/24`, а не `/0`.
2. Если точного маршрута нет — используется **маршрут по умолчанию** (`0.0.0.0/0`). Это «последняя надежда», обычно указывает на провайдера.

## Пример работы (простая схема)

Представим: ваш компьютер (192.168.1.10) отправляет пакет на Google DNS (8.8.8.8).

1. Компьютер вычисляет: `8.8.8.8` НЕ в его сети `/24` → нужно отправить через **шлюз по умолчанию** (192.168.1.1).
2. Компьютер создаёт IP-пакет (src=192.168.1.10, dst=8.8.8.8).
3. Компьютер отдаёт пакет канальному уровню, который создаёт Ethernet-фрейм с dst MAC = MAC-адрес шлюза (а не 8.8.8.8!).
4. Шлюз (роутер) получает фрейм, извлекает IP-пакет, видит dst=8.8.8.8.
5. Роутер смотрит в свою таблицу: для 8.8.8.8 нет точного маршрута → отправляет по default gateway (адрес провайдера 95.64.12.1) через WAN-интерфейс.
6. Провайдерский маршрутизатор делает то же самое, передавая пакет дальше.
7. На каждом перескоке (hop) MAC-адрес получателя в канальном уровне меняется, а IP-адрес назначения (8.8.8.8) остаётся неизменным.

Это называется **сквозным принципом** (end-to-end) — вся сложность маршрутизации скрыта внутри сети, а конечные узлы видят только начальную и конечную точку.

# Связь с канальным уровнем

Очень часто путают IP и MAC-адреса, но их роли принципиально разные:

| Характеристика | MAC-адрес (L2) | IP-адрес (L3) |
|----------------|----------------|---------------|
| Длина | 48 бит (6 байт) | 32 бита (4 байта) |
| Формат | `AA:BB:CC:11:22:33` | `192.168.1.10` |
| Кто назначает | Производитель оборудования | Администратор или DHCP |
| Меняется ли | Обычно нет (бывает программно) | Может меняться хоть каждый час |
| Где используется | Внутри одной локальной сети (broadcast-домена) | Глобально, между сетями |
| Какой уровень | Канальный | Сетевой |

**Как они работают вместе:**
1. Компьютер A знает IP-адрес компьютера B (8.8.8.8), но не знает его MAC.
2. Чтобы отправить фрейм внутри своей сети шлюзу, компьютер A использует **ARP-протокол** (Address Resolution Protocol):
   - A отправляет broadcast-запрос: «Кто имеет IP 192.168.1.1? Сообщи свой MAC».
   - Шлюз отвечает: «Я, мой MAC AA:BB:CC:11:22:33».
3. Теперь A может создать Ethernet-фрейм: dst MAC = AA:BB:CC:11:22:33, внутри — IP-пакет к 8.8.8.8.
4. На следующем линке (между роутером и провайдером) будет новый ARP-запрос и новый MAC-адрес получателя.

> **Золотое правило:** MAC-адрес меняется на каждом линке, IP-адрес остаётся неизменным от отправителя до получателя.
