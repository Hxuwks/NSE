## Прикладной уровень: общая концепция

**Прикладной уровень (Application Layer)** — самый верхний уровень модели сетевого взаимодействия. Он обеспечивает интерфейс между пользовательскими приложениями и сетью. Именно здесь «живут» все знакомые каждому протоколы: HTTP для веб-серфинга, DNS для преобразования имён, SSH для удалённого управления, FTP для передачи файлов.

**Ключевая идея:** прикладной уровень скрывает от пользователя всю сложность нижележащих уровней (TCP, IP, Ethernet). Когда вы открываете браузер, вы не думаете о сегментах, пакетах и кадрах — вы просто вводите адрес. Всю «грязную» работу по установке соединения, разбивке данных, маршрутизации и восстановлению ошибок берут на себя нижние уровни.

> **Важно:** протоколы прикладного уровня используют транспортный уровень (TCP или UDP) как средство доставки. Выбор транспорта определяет, будет ли соединение надёжным (TCP) или быстрым, но негарантированным (UDP).

## 1. HTTP (HyperText Transfer Protocol) — протокол передачи гипертекста

**HTTP** — фундаментальный протокол Всемирной паутины (WWW). Он определяет, как веб-браузеры и веб-серверы обмениваются информацией: запросами и ответами.

### Основные характеристики:

- **Работает поверх TCP** (обычно порт 80). Это обеспечивает надёжную доставку запросов и ответов.
- **Модель «клиент-сервер»:** клиент (браузер) инициирует соединение и отправляет запрос; сервер обрабатывает его и возвращает ответ.
- **Протокол без сохранения состояния (stateless):** сервер не запоминает предыдущие запросы клиента. Каждый запрос выполняется независимо. Для поддержания сеансов (например, корзина покупок) используются механизмы поверх HTTP: **cookies**, сессионные идентификаторы в URL или заголовках.
- **Текстовый протокол:** запросы и ответы представляют собой читаемые человеком строки (заголовки и тело сообщения).

### Структура HTTP-запроса:

```text
МЕТОД /ресурс/путь HTTP/ВЕРСИЯ
Заголовок1: значение
Заголовок2: значение
Пустая строка
Тело запроса (например, данные формы или JSON)
```

**Пример запроса (получение главной страницы):**
```http
GET /index.html HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0
Connection: keep-alive

```

### Структура HTTP-ответа:

```text
HTTP/ВЕРСИЯ КОД_СТАТУСА СООБЩЕНИЕ
Заголовок1: значение
Заголовок2: значение
Пустая строка
Тело ответа (HTML-страница, JSON, изображение и т.д.)
```

**Пример ответа (успех):**
```http
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 1234

<!DOCTYPE html>
<html>
<head><title>Пример</title></head>
<body>...</body>
</html>
```

### Основные методы HTTP:

| Метод | Назначение | Безопасность (идемпотентность) |
|-------|------------|--------------------------------|
| `GET` | Получение ресурса | Да (не изменяет данные) |
| `POST` | Отправка данных на сервер (создание ресурса) | Нет |
| `PUT` | Полная замена ресурса | Да |
| `DELETE` | Удаление ресурса | Да |
| `PATCH` | Частичное обновление ресурса | Нет |
| `HEAD` | Получение только заголовков (без тела) | Да |

### Коды состояния HTTP (группы):

| Диапазон | Назначение | Примеры |
|----------|------------|---------|
| **1xx** | Информационные | 100 Continue |
| **2xx** | Успех (Success) | **200 OK**, 201 Created |
| **3xx** | Перенаправление (Redirection) | **301** Moved Permanently, **302** Found, 304 Not Modified |
| **4xx** | Ошибка клиента | **400** Bad Request, **401** Unauthorized, **403** Forbidden, **404** Not Found |
| **5xx** | Ошибка сервера | **500** Internal Server Error, 502 Bad Gateway, 503 Service Unavailable |

### Версии HTTP (историческая справка):

- **HTTP/1.0** (1996): каждое соединение — один запрос-ответ. Неэффективно.
- **HTTP/1.1** (1997): **Keep-Alive** — одно соединение на несколько запросов, конвейеризация. Широко используется до сих пор.
- **HTTP/2** (2015): мультиплексирование, сжатие заголовков, приоритеты. Быстрее.
- **HTTP/3** (2022): работает поверх **QUIC** (UDP), устраняет проблемы «задержки начала соединения» (head-of-line blocking) на уровне TCP.

## 2. HTTPS (HTTP Secure) — защищённый HTTP

**HTTPS** — это HTTP, работающий поверх криптографического протокола **TLS** (Transport Layer Security, ранее SSL). Он обеспечивает три ключевых свойства:

1. **Шифрование** — третья сторона не может прочитать передаваемые данные.
2. **Аутентификация сервера** — клиент может удостовериться, что он общается именно с тем сервером, которому доверяет (через цифровые сертификаты).
3. **Целостность** — данные не могут быть изменены незаметно в процессе передачи.

### Как работает HTTPS (упрощённо):

1. Клиент устанавливает TCP-соединение с сервером на порту **443** (стандартный порт HTTPS).
2. Начинается **рукопожатие TLS** (TLS handshake):
   - Клиент и сервер договариваются о версии TLS и наборе шифров.
   - Сервер передаёт свой **цифровой сертификат** (содержит открытый ключ, подписанный центром сертификации).
   - Клиент проверяет сертификат (не истёк ли срок, подписан ли доверенным центром, совпадает ли домен).
   - Клиент генерирует случайный сессионный ключ, шифрует его открытым ключом сервера и отправляет.
3. Начиная с этого момента все данные шифруются сессионным ключом (симметричное шифрование, быстрее асимметричного).
4. Поверх защищённого канала передаются обычные HTTP-запросы и ответы.

> **Итог:** HTTPS = HTTP + TLS. Для браузера и сервера это прозрачно — они по-прежнему общаются через HTTP, просто весь трафик шифруется на транспортном уровне.

### Почему HTTPS важен:

- **Конфиденциальность:** пароли, номера карт, личные сообщения не передаются открыто.
- **Защита от подмены:** невозможно осуществить атаку Man-in-the-Middle без обнаружения.
- **SEO и доверие:** современные браузеры помечают HTTP-сайты как «незащищённые».

## 3. DNS (Domain Name System) — система доменных имён

**DNS** — это «телефонная книга» Интернета. Она преобразует человекочитаемые доменные имена (например, `example.com`) в машинные IP-адреса (например, `93.184.216.34`). Без DNS мы бы запоминали для каждого сайта числовой адрес — что крайне неудобно.

### Основные характеристики:

- **Работает поверх UDP** (порт 53) для запросов (быстро, не требует установки соединения) и TCP (для больших ответов, например, зонных передач).
- **Распределённая иерархическая система:** нет единой базы данных. Вся структура разбита на зоны, за каждую зону отвечают свои серверы.
- **Кэширование:** результаты запросов сохраняются на клиентах, на локальных DNS-резолверах и на вышестоящих серверах. Это резко снижает нагрузку на корневые серверы и ускоряет работу.

### Иерархия DNS (сверху вниз):

```text
. (корневой уровень) → 13 групп корневых серверов (A–M)
  │
  ├── .com (зона верхнего уровня — TLD)
  │     └── example.com (вторичный домен)
  │           └── www.example.com (поддомен/запись A)
  ├── .org
  ├── .ru
  │     └── yandex.ru
  └── ...
```

**Каждый домен выше делегирует зону нижестоящим серверам.** Например, корневые серверы знают, где находятся DNS-серверы для зоны `.com`, серверы `.com` знают, где находится `example.com`, и так далее.

### Типы DNS-записей (важнейшие):

| Тип записи | Назначение | Пример |
|------------|------------|--------|
| **A** | IPv4-адрес узла | `example.com → 93.184.216.34` |
| **AAAA** | IPv6-адрес узла | `example.com → 2606:2800:220:1:248:1893:25c8:1946` |
| **CNAME** | Каноническое имя (псевдоним) | `www.example.com → example.com` |
| **MX** | Почтовый обменник (Mail Exchange) | `example.com → mail.example.com (приоритет 10)` |
| **NS** | Сервер имён (Authoritative NS) | `example.com → ns1.example.com` |
| **TXT** | Произвольный текст (SPF, DKIM, проверки домена) | `example.com → "v=spf1 include:_spf.google.com ~all"` |
| **PTR** | Обратная запись (IP → имя) | `34.216.184.93.in-addr.arpa → example.com` |
| **SOA** | Начало авторитарной зоны (Start of Authority) | содержит параметры зоны (серийный номер, TTL и т.д.) |

### Процесс разрешения доменного имени (рекурсивный запрос):

```
1. Браузер → локальный DNS-резолвер (обычно 8.8.8.8 или ваш провайдер)
2. Локальный резолвер (если нет в кэше) → корневой DNS-сервер
3. Корневой сервер → отправляет на TLD-сервер для .com
4. TLD-сервер .com → отправляет на авторитарный DNS-сервер для example.com
5. Авторитарный сервер example.com → возвращает A-запись (IP-адрес)
6. Локальный резолвер возвращает IP-адрес браузеру
7. Браузер устанавливает соединение по полученному IP-адресу
```

## 4. FTP (File Transfer Protocol) — протокол передачи файлов

**FTP** — один из старейших интернет-протоколов (RFC 959, 1985). Он предназначен для передачи файлов между клиентом и сервером: загрузки (upload) и скачивания (download).

### Особенности:

- **Работает поверх TCP** (надёжная доставка).
- **Два канала:** управляющий (команды) и данных.
  - **Управляющий канал:** порт **21** (TCP). Клиент отправляет команды (`USER`, `PASS`, `LIST`, `RETR`, `STOR`, `QUIT`), сервер отвечает кодами состояния (аналогично HTTP).
  - **Канал данных:** отдельное соединение, которое открывается для передачи содержимого файлов или списков директорий. Может быть активным (сервер подключается к клиенту) или пассивным (клиент подключается к серверу) — это важно для работы через NAT/фаерволы.

### Режимы FTP:

- **Активный режим:** клиент сообщает серверу свой IP и случайный порт, сервер подключается к клиенту для передачи данных. Проблемы: клиент может быть за NAT/фаерволом.
- **Пассивный режим (PASV):** сервер сообщает клиенту IP и порт, к которому клиент подключается сам. Рекомендуется для современных сетей.

### Безопасность FTP:

- **Обычный FTP** не шифрует данные (логины, пароли, сами файлы передаются открыто).
- **FTPS (FTP Secure):** FTP поверх SSL/TLS (аналогично HTTPS, но для FTP).
- **SFTP (SSH File Transfer Protocol):** работает поверх SSH (порт 22), а не FTP-стек. Шифрует всё. На самом деле SFTP — отдельный протокол, не являющийся расширением FTP.

> **Важно:** в современном мире обычный FTP считается небезопасным. Предпочтение отдаётся **SFTP** или **SCP** (через SSH).

## 5. SSH (Secure Shell) — безопасная командная оболочка

**SSH** — протокол для удалённого управления системами через защищённый канал. Он полностью заменил небезопасные протоколы **Telnet**, **rlogin** и **rsh**, которые передавали пароли и данные открыто.

### Основные функции SSH:

1. **Удалённый доступ к командной оболочке** (shell) на сервере.
2. **Безопасная передача файлов** через SFTP и SCP.
3. **Туннелирование (порт-форвардинг)** — перенаправление TCP-соединений через зашифрованный SSH-канал (к примеру, безопасный доступ к базе данных, не имеющей собственного шифрования).
4. **Запуск удалённых команд** без интерактивного сеанса.

### Как работает SSH (аутентификация и шифрование):

1. Клиент инициирует TCP-соединение с сервером на порту **22**.
2. Сервер отправляет свой публичный ключ (host key) и предлагает клиенту выбрать метод аутентификации.
3. Начинается согласование алгоритмов шифрования, обмен ключами по Диффи-Хеллману (устанавливается общий секретный ключ).
4. Вся дальнейшая передача данных шифруется сессионным ключом.
5. Аутентификация клиента может быть выполнена:
   - **По паролю** (просто, но уязвимо к подбору).
   - **По ключу** (рекомендуемый способ): клиент предоставляет публичный ключ, сервер проверяет подпись, созданную закрытым ключом клиента. Это более безопасно и удобно при автоматизации.
   - Другие методы: Kerberos, аутентификация по сертификатам.

### Основные команды SSH (клиент OpenSSH):

```bash
# Подключиться к серверу
ssh user@hostname

# Подключиться с указанием порта (если нестандартный)
ssh -p 2222 user@example.com

# Запустить одну команду на сервере и выйти
ssh user@example.com "ls -la /var/log"

# Копирование файлов через SCP (Secure Copy)
scp file.txt user@example.com:/home/user/

# Копирование файлов через SFTP (интерактивная сессия)
sftp user@example.com

# Туннелирование (порт-форвардинг): локальный порт 8080 → remote:80
ssh -L 8080:remote:80 user@example.com
```

### Преимущества SSH над Telnet и другими старыми протоколами:

| Протокол | Шифрование | Аутентификация | Современный статус |
|----------|------------|----------------|-------------------|
| **Telnet** | Нет | Пароль в открытом виде | Считается небезопасным, не рекомендуется к использованию |
| **rsh / rlogin** | Нет | На основе IP и .rhosts | Устарел, опасен |
| **SSH** | Да (симметричное шифрование) | Ключи / пароль (защищённо) | Стандарт де-факто |

## Сравнительная таблица протоколов прикладного уровня

| Протокол | Порт (стандартный) | Транспорт | Назначение | Особенности |
|----------|:------------------:|:----------|------------|-------------|
| **HTTP** | 80 | TCP | Передача гипертекста | Незащищённый, stateless, текстовый |
| **HTTPS** | 443 | TCP (TLS) | Защищённая передача гипертекста | Шифрование, аутентификация, целостность |
| **DNS** | 53 | UDP (и TCP) | Разрешение доменных имён | Распределённая иерархия, кэширование |
| **FTP** | 21 (управление), 20 (данные) | TCP | Передача файлов | Два канала, активный/пассивный режимы. Небезопасен без TLS |
| **SSH** | 22 | TCP | Удалённое управление, передача файлов | Полное шифрование, аутентификация по ключам, туннелирование |

## Резюме (основные выводы)

1. **Прикладной уровень** — это интерфейс между приложениями и сетью. Здесь определяются форматы запросов, ответов и правила поведения.
2. **HTTP/HTTPS** — основа современного интернета. HTTP — текстовый и незащищённый, HTTPS — тот же HTTP, но поверх TLS.
3. **DNS** — критическая инфраструктура, преобразующая имена в IP-адреса. Работает по иерархическому принципу с кэшированием.
4. **FTP** — старый протокол передачи файлов с отдельными каналами управления и данных. Из-за отсутствия шифрования вытесняется SFTP.
5. **SSH** — стандарт для защищённого удалённого доступа и передачи файлов в UNIX-среде. Шифрует всё, поддерживает аутентификацию по ключам.
6. Все перечисленные протоколы (кроме DNS, частично использующего UDP) работают поверх **TCP**, обеспечивая надёжную доставку.
