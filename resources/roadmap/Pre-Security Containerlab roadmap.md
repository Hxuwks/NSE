## Этап 0. Подготовка хоста (4–8 ч)

- Установить Docker (движок + compose plugin по документации дистрибутива).
- Установить Containerlab по [официальной инструкции](https://containerlab.dev/install/).
- Проверить: `docker run hello-world`, `containerlab version`.
- Настроить удобный запуск: пользователь в группе `docker` или обёртка с `sudo` — зафиксировать в заметках, что выбрал.

**Артефакт:** скрин или лог успешной установки + краткий README «как воспроизвести».

---

## Этап 1. Первая топология и жизненный цикл (6–10 ч)

- Разобрать минимальный `topology.yml`: `name`, `topology.nodes`, `topology.links`.
- Команды: `containerlab deploy -t <file>`, `inspect`, `destroy`, опционально `graph`.
- Поднять два узла `linux` (например `alpine`/`debian`), соединить линком, зайти: `docker exec -it <name> bash/sh`.
- Назначить адреса вручную (`ip addr add ...`, `ip link set up`), проверить `ping`.

**Артефакт:** репозиторий с `01-two-hosts.clab.yml` + README (схема, команды, вывод `ping`).

---

## Этап 2. Маршрутизация: FRR и статика (12–16 ч)

- Добавить kind `frr` (или образ FRR из документации Containerlab для твоей версии).
- Топология: три маршрутизатора треугольником или «два клиента — два маршрутизатора — хаб».
- Конфигурация через `vtysh`: hostname, интерфейсы, **статические маршруты** к удалённым подсетям.
- Проверка: `show ip route`, `show int`, ping между всеми конечными хостами.
- Сохранение конфига: понять, как в твоём сценарии переживает перезапуск (volume, `frr.conf`, скрипт при старте).

**Артефакт:** `02-static-routing.clab.yml` + экспорт `frr.conf` или скрипт наката конфигурации.

---

## Этап 3. DHCP и DNS (8–12 ч)

- Выделить узел **linux** как сервер: **dnsmasq** или **Kea DHCP**.
- Один или два клиента получают адрес по DHCP; проверить аренду (`journalctl`, файлы lease, `ip addr`).
- Минимум DNS: разрешение имени тестового хоста (через dnsmasq `address=/` или отдельный Unbound — по желанию).

**Артефакт:** `03-dhcp-dns.clab.yml` + описание пула и проверки с клиента.

---

## Этап 4. NAT и «внешняя» сеть лаборатории (8–12 ч)

- Схема: внутренняя подсеть → **Linux-шлюз** с `net.ipv4.ip_forward=1`.
- **SNAT/PAT:** `iptables -t nat -A POSTROUTING -o <uplink> -j MASQUERADE` (или эквивалент **nftables**).
- «Интернет» в лаборатории: отдельный linux-узел с публичной подсетью лаборатории или эмуляция провайдера без реального выхода.
- Проверить с внутреннего хоста доступ наружу и обратные маршруты.

**Артефакт:** `04-nat-gateway.clab.yml` + таблица правил NAT в README.

---

## Этап 5. Фильтрация (аналог «простых ACL») (8–12 ч)

- На шлюзе или отдельном «firewall»-контейнере: цепочки **filter**, политика по умолчанию DROP/ACCEPT (осознанно).
- Правила: разрешить SSH с админской подсети, закрыть доступ из гостевой подсети к серверной, логировать дропы.
- Проверить `nft list ruleset` или `iptables -L -n -v`.

**Артефакт:** `05-firewall-rules.clab.yml` + таблица правил (куда/откуда/протокол/действие).

---

## Этап 6. Сегментация без классического L2 (6–10 ч)

- Спроектировать «два отдела»: две подсети, разные `links` в YAML, маршрутизация на FRR.
- Документировать, как это соотносится с идеей VLAN (логическая изоляция), и явно написать, что **802.1Q trunk** ты закрепишь отдельно (железо / GNS3 / EVE-NG).

**Артефакт:** схема + `06-segments.clab.yml`.

---

## Этап 7. Диагностика (4–8 ч)

- `tcpdump` / `tshark` на линке между узлами; базовые фильтры (ICMP, TCP port).
- Сохранить **pcap** и коротко описать, что видно (эхо-запрос/ответ, TTL).

**Артефакт:** `07-capture.pcap` + 5–10 строк разбора в README.

---

## Финальный мини-проект Pre-Security (интеграция)

Собрать **одну** топологию, которая включает:

- не менее трёх подсетей;
- FRR (или два FRR) со статикой;
- DHCP для хотя бы одной подсети;
- NAT на выходе;
- минимум пять осмысленных firewall-правил;
- README с диаграммой и инструкцией «как поднять с нуля».

**Артефакт:** папка `final-presec-lab/` в Git с полным воспроизводимым сценарием.
